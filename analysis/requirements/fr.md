# Функциональные требования

| ID | Формулировка | Приоритет | Ссылка на источник |
|----|--------------|-----------|---------------------|
| FR-01 | Приложение должно отображать список доступных слотов на ближайшие 7 дней от текущей даты | Must have | R-027: Дефолтный список слотов — 7 дней |
| FR-02 | Приложение должно предоставлять фильтр по датам для просмотра расписания за пределами 7-дневного окна | Should have | R-027: Более длинный период доступен через фильтр дат |
| FR-03 | Приложение должно отображать для каждого слота: время начала, зону/формат тренировки, ФИО инструктора, количество свободных мест | Must have | Расписание на неделю вперёд: время, зона, инструктор, места |
| FR-04 | Приложение должно позволять клиенту записаться на конкретный слот | Must have | Клиент должен видеть свободные слоты и записаться |
| FR-05 | При записи приложение должно предоставлять выбор: использование собственного снаряжения (скальники, страховка) или прокат | Should have | Выбирал, со своими скальниками и страховкой или ему дать напрокат |
| FR-06 | Приложение должно отображать стоимость проката снаряжения в соответствии с тарифом из бэкенда | Should have | R-015: контракт API включает прокатный тариф на снаряжение |
| FR-07 | Приложение должно проверять свободный прокатный фонд снаряжения при выборе опции проката | Should have | R-015: контракт API включает свободный прокатный фонд снаряжения |
| FR-08 | Приложение должно обрабатывать отказ при попытке бронирования занятого слота и отображать соответствующее сообщение пользователю | Must have | Граница закрывается контрактом API — корректно обрабатывает отказ при отсутствии мест |
| FR-09 | Приложение должно позволять клиенту отменять собственную запись на тренировку | Should have | Иногда люди отменяются |
| FR-10 | При отмене записи приложение должно отображать временное ограничение (за 10 минут до начала — проблемная зона) | Should have | Если за 10 минут до начала — это уже проблема, место простаивает |
| FR-11 | При отмене записи в «проблемной зоне» приложение должно уведомлять клиента о последствиях | Could have | Надо как-то это ограничить, но чтобы клиентов не злить |
| FR-12 | После завершения тренировки приложение должно предоставлять клиенту форму оценки инструктора | Should have | Хочу, чтобы после тренировки клиент мог поставить оценку инструктору |
| FR-13 | Приложение должно отображать статус «Отменена скалодромом» для броней, отменённых организатором, с указанием причины | Must have | Бронь не удаляется, а получает статус «Отменена скалодромом» с указанием причины |
| FR-14 | Приложение должно запрещать повторную запись на слот, отменённый скалодромом | Must have | Повторная запись на отменённый слот запрещена |
| FR-15 | Приложение должно получать push-уведомления об отмене тренировки скалодромом | Must have | Клиенту приходит push-уведомление |
| FR-16 | Приложение должно отправлять push-напоминания клиентам о предстоящих тренировках | Could have | Уведомления, чтобы напоминать про запись |
| FR-17 | Приложение должно отображать empty state при отсутствии доступных тренировок в выбранном периоде | Must have | Если расписания на ближайшие дни нет, показывается понятный empty state |
| FR-18 | Приложение должно потреблять данные о слотах, зонах/форматах и инструкторах из существующего бэкенда через API | Must have | Слоты, зоны/форматы и инструкторы поступают из существующего бэкенда через API |
| FR-19 | Приложение должно отображать адрес скалодрома из данных бэкенда | Should have | R-015: контракт API включает адрес скалодрома |
| FR-20 | Приложение должно поддерживать мобильную ориентацию и адаптивный интерфейс для использования на смартфонах | Must have | Пользоваться будут в основном с телефона |
